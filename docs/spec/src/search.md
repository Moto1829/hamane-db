# 検索

## API

```rust
let hits: Vec<SearchHit> = col.search(&query)
    .k(10)                            // 取得件数 (既定 10)
    .filter(Filter::eq("lang", "ja")) // メタデータフィルタ (任意)
    .ef(128)                          // HNSW 探索幅の上書き (任意、既定 64)
    .run()?;

pub struct SearchHit {
    pub id: Id,            // u64
    pub score: f32,        // 意味はメトリック依存 (下表)
    pub metadata: Metadata,
}
```

結果は**近い順**に並びます。

| Metric | score | 並び順 |
|---|---|---|
| `L2` | ユークリッド距離 | 昇順 |
| `Cosine` | コサイン類似度 | 降順 |
| `Dot` | 内積 | 降順 |

## 実行モデル

検索はスナップショット上で実行されます。開始時点の memtable と
セグメント集合が対象になり、検索中の書き込み・フラッシュ・コンパクションの
影響を受けません (ブロックもしません)。

データソースごとの実行方式:

| ソース | 方式 | 正確性 |
|---|---|---|
| memtable (未フラッシュ分) | Flat (全走査) | 正確 |
| セグメント (HNSW なし = 小規模) | Flat | 正確 |
| セグメント (HNSW あり) | HNSW | **近似** |

各ソースの上位 k を距離順にマージして最終結果を返します。

## 一貫性の保証

- **削除済み・上書き済みレコードは決して結果に現れない**。
  複数セグメントにまたがる同一 id は、最も新しい書き込みだけが有効
  (newest-wins)。この判定は走査時に行われるため、Flat 経路では
  古い値が新しい値を押し出すこともない
- 挿入・削除は**即座に**検索へ反映される (インデックス構築を待たない)
- 同距離のタイは id 昇順で決定的に順序づけられる (Flat 経路)

## 近似性 (HNSW) について

HNSW を持つセグメントに対する検索は近似です。再現率は `ef` で制御します:

- `ef` は「探索中に保持する候補数」。大きいほど再現率が上がり遅くなる。
  実効値は `max(ef, k)`
- 既定パラメータ (`m=16, ef_construction=200, ef=64`) での実測は
  SIFT1M で recall@10 = 0.977 ([ベンチマーク](benchmarks.md))
- `ef` をレコード数以上にすると実質的に全探索と一致する

HNSW が構築されるのはフラッシュされたセグメントのうち行数が
`hnsw_min_rows` (既定 1024) 以上のものだけです。それ未満のセグメントと
memtable は常に正確検索されます。

## フィルタ付き検索の実行戦略

フィルタがある場合、セグメントごとに選択率 (フィルタ一致率) を
サンプリング推定し、実行戦略を自動選択します:

| 選択率 | 戦略 | 内容 |
|---|---|---|
| < 5% | **pre-filter** | メタデータを走査して一致行だけ距離計算 (正確) |
| ≥ 5% | **post-filter** | HNSW 探索中に不一致候補を棄却。探索幅を最大 4 倍に自動拡大 |

注意: post-filter は近似のため、一致レコードが存在しても k 件に
満たない結果が返る可能性があります。厳密性が必要な場合は `ef` を
大きくしてください。

## SQ8 量子化による高速化

`StoreOptions { sq8: true, .. }` で有効化 (既定 off)。フラッシュ時に
ベクトルを u8 に量子化した `vectors_sq8.bin` も書き、HNSW 探索の
距離計算を整数演算で行います。

- 探索は SQ8 距離で上位 k×4 件を取り、f32 の正確な距離で再ランクして
  上位 k を返す (2 段階検索)。**返される score は常に正確な f32 距離**
- 量子化は全次元共通の min/max (グローバルスケール)
- フィルタなしの HNSW 経路にのみ適用される。pre-filter (Flat) 経路は
  常に f32

## 実行の並列性

複数セグメントを持つ collection では、セグメントごとの探索が
Database 全体で共有する常駐スレッドプールで並列実行されます
(セグメント 1 個以下では逐次)。並列度は
[`StoreOptions.search_threads`](configuration.md#storeoptions)
で指定します (0 = 自動 = 論理コア数、1 = 逐次)。

## 計算量の目安

| 操作 | 計算量 |
|---|---|
| Flat 検索 (memtable / 小セグメント) | O(n log k) |
| HNSW 検索 | O(ef · log n) 相当 (経験的) |
| 点参照 `get(id)` | O(セグメント数 × log n) |
| `len()` | O(1) (書き込み時に差分維持) |
