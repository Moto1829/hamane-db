# 制限事項と既知の制約

v0.1 時点の制約です。括弧内は対応予定のタスク番号 (リポジトリ todos/)。

## 使用上の制約

- **単一プロセス専用**。同じ DB ディレクトリを複数プロセスから開くことは
  想定しておらず、ロック機構もありません。複数プロセスで開いた場合の
  動作は未定義です (データ破損の可能性あり)
- **ID は u64 のみ**。文字列 ID は未対応 (601)
- Collection の `dim` / `metric` は作成後変更不可。変更するには
  新しい Collection を作ってデータを入れ直す
- トランザクション (複数操作の原子性) はない。原子性の単位は
  1 回の `upsert` / `delete` / `upsert_batch`
- メタデータの値はスカラー 4 型のみ (配列・ネストは不可)

## 性能上の制約

- **フラッシュ・コンパクションが書き込みをブロックする** (504)。
  これらは書き込みスレッド上で同期実行され、HNSW 構築を含むため
  大規模 memtable では分単位で書き込みが停止する。当面の回避策:
  `flush_threshold_bytes` を下げて 1 回あたりのフラッシュを小さくする、
  または書き込みの少ない時間帯に明示的に `flush()` する
- **HNSW 構築は単一スレッド** (501)。SIFT1M (100 万 × 128 次元) で
  約 24 分。挿入自体 (WAL + memtable) は高速 (約 30 万 rec/s)
- `len()` は全セグメント走査を伴い O(総行数) (503)
- 検索のソース (セグメント) 間は逐次実行 (503)
- コンパクションは全セグメント統合 (full merge) のため、
  書き込み総量が大きい (506)

## 検索の制約

- HNSW 経路は近似であり、recall = 1.0 は保証されない
  (実測は [ベンチマーク](benchmarks.md))
- post-filter 経路 (選択率 ≥ 5% のフィルタ) では、一致レコードが
  存在しても k 件未満の結果になることがある
- `delete()` の戻り値 (存在したか) は並行書き込みがある場合に
  厳密でないことがある (507)

## 運用の制約

- `StoreOptions` は永続化されず、open のたびに指定が必要。
  異なる `hnsw` パラメータで開いても既存セグメントの HNSW は
  再構築されない (次のフラッシュ/コンパクションから適用)
- バックアップ機構はない。安全にコピーするには DB を閉じた状態で
  ディレクトリ全体をコピーする
- 内部ロックが poisoned になった場合 (書き込み中の panic 後) は
  以後の操作が panic する。プロセス再起動で復旧する
