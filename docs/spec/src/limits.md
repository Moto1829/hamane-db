# 制限事項と既知の制約

v0.3 (M7 完了) 時点の制約です。

## 使用上の制約

- **単一プロセス専用**。同じ DB を二重に開こうとすると `Locked` エラーに
  なります (unix: flock による排他。非 unix はベストエフォート)
- Collection の `dim` / `metric` は作成後変更不可。変更するには
  新しい Collection を作ってデータを入れ直す
- トランザクション (複数操作の原子性) はない。原子性の単位は
  1 回の `upsert` / `delete` / `upsert_batch`
- メタデータの値はスカラー 4 型のみ (配列・ネストは不可)。
  `_ext_id` キーは文字列 ID 用に予約
- u64 ID と文字列 ID を混在させる場合、u64 側は 2^63 未満を使うこと
  (文字列 ID の内部採番領域と衝突するため)

## 性能上の制約

- HNSW の並列構築 (`build_threads` ≥ 2、既定) ではグラフが実行ごとに
  変わる (非決定)。再現性が必要な場合は `build_threads: 1` を指定する
  (構築は約 5 倍遅くなる)
- 書き込みが極端に速く、フラッシュ (HNSW 構築) が追いつかない場合、
  active memtable が閾値の 4 倍に達した時点で書き込みが
  フラッシュ完了を待つ (backpressure。メモリ保護のための設計)
- SQ8 の距離カーネルはスカラー実装 (u8 SIMD は将来最適化)

## 検索の制約

- HNSW 経路は近似であり、recall = 1.0 は保証されない
  (実測は [ベンチマーク](benchmarks.md))
- post-filter 経路 (選択率 ≥ 5% のフィルタ) では、一致レコードが
  存在しても k 件未満の結果になることがある。厳密性が必要なら `ef` を
  上げる
- SQ8 有効時、量子化誤差により HNSW 探索の候補選択がわずかに変わり得る
  (最終 score は f32 再ランクで常に正確)

## 運用の制約

- `StoreOptions` は永続化されず、open のたびに指定が必要。
  異なる `hnsw` / `sq8` 設定で開いても既存セグメントは再構築されない
  (次のフラッシュ / コンパクションから適用)
- HTTP サーバ (hamane-server) に認証・TLS はない (リバースプロキシ前提)
- `Database::backup(dest)` のコピー中は書き込みが待たされる
  (コピーは純粋な I/O で短時間)
- 内部ロックが poisoned になった場合 (書き込み中の panic 後) は
  以後の操作が panic する。プロセス再起動で復旧する

## v0.1 から解消された制約

| v0.1 の制約 | 解消 (v0.2) |
|---|---|
| フラッシュ中は書き込みが分単位で停止 | バックグラウンド実行 (p99 8µs) |
| HNSW 構築が単一スレッド (1M = 24 分) | 並列構築 (1M = 5 分) |
| `len()` が O(総行数) | O(1) (差分維持) |
| セグメント間の検索が逐次 | スレッド並列 |
| ID は u64 のみ | 文字列 ID 対応 |
| コンパクションが常に full merge | 自動は部分マージ (universal 風) |
| 多重 open が未定義動作 | Locked エラー (v0.3 / M7) |
| バックアップ機構なし | Database::backup (v0.3 / M7) |
| SQ8 距離がスカラーのみ | NEON で f32 比 4x (v0.3 / M7) |
